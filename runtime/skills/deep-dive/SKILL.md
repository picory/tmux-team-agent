# Deep Dive — 요구사항 인터뷰 및 스펙 작성

## 개요

모호한 작업을 집중 인터뷰로 명확히 하고, 기존 문서를 업데이트하거나 새 스펙을 작성한다.
- 기본 출력 파일: `spec-<topic>.md`
- 관련 문서가 있으면 **신규 작성보다 업데이트를 기본**으로 한다
- 질문은 적게, 더 깊게, 이전 답변을 누적해서 발전시킨다

---

## Step 1: 워크스페이스 스캔

```bash
ls spec-*.md *-spec.md blueprint-*.md *PRD*.md *기획*.md *설계*.md 2>/dev/null || echo "없음"
cat CLAUDE.md 2>/dev/null | head -20 || true
cat README.md 2>/dev/null | head -20 || true
```

관련 문서가 있으면 내용과 섹션 구조를 요약한다.

---

## Step 2: 업데이트 vs 신규 확인

관련 문서가 있으면 인터뷰 전에 확인한다:
- 강력한 후보 1개: "기존 문서 업데이트(권장)" vs "새 문서 작성" 선택
- 후보 여러 개: 가장 적합한 것을 추천하고 선택지 제공
- 사용자가 "new"를 명시하지 않으면 업데이트를 기본으로 한다

---

## Step 3: 집중 인터뷰

아래 5개 영역을 순서대로 파고든다. 한 번에 최대 2개 질문.
이전 답변을 반드시 누적해서 참조한다.

1. **핵심 행동**: 정확히 무엇을 언제 어떻게 하는가
2. **기술 제약**: 사용 가능한 API, 데이터 형식, 시스템 한계
3. **UX / 인터페이스**: 사용자 접점, 입출력 형태
4. **트레이드오프**: 정확도 vs 속도, 자동화 vs 수동 개입
5. **실패 모드**: 무엇이 잘못될 수 있고 어떻게 감지/복구하는가

최대 4라운드 후 미확인 항목은 가정으로 처리하고 문서 작성으로 전환한다.

---

## Step 4: 문서 작성 / 업데이트

**업데이트**: 변경된 섹션만 `Edit`으로 정밀 수정. 관련 없는 내용은 건드리지 않는다.

**신규 작성**: `spec-<topic>.md` 구조:

```
# Spec: <제목>

## 배경 및 목표
## 범위 (In / Out of Scope)
## 입력 / 출력
## 핵심 행동 상세
## 기술 제약
## 실패 처리
## 열린 질문 / 가정
## 다음 액션
```

---

## Step 5: 완료 보고

```
완료
  파일: <경로>
  작업: <업데이트 | 신규 작성>
  열린 질문: <있으면 나열>
  다음 액션: <1-2개>
```
